## Задание

Сравнить алгоритмы приближённого поиска ближайших соседей (Approximate Nearest Neighbor, ANN) на реальных векторных данных. Для каждого алгоритма проверить несколько конфигураций параметров и оценить компромисс между точностью (Recall100), скоростью запроса (QPS) и размером индекса.

---

## Датасет

Использовалась предобученная модель **word2vec-ruscorpora-300** из библиотеки `gensim` - векторные представления слов русского языка, обученные на корпусе RNC (Национальный корпус русского языка). Каждое слово представлено вектором размерностью **300**.

Для чистоты эксперимента датасет был отфильтрован: оставлены только существительные (слова с тегом `_NOUN`). Итоговый корпус составил порядка **185 тысяч слов**.

Из корпуса было выбрано 10 000 запросных слов, а индекс для сравнения строился на подмножестве из 50 000 векторов (включающем все 10 000 запросных слов). Для каждого запросного слова была вычислена точная «ground truth» - 100 ближайших соседей методом полного перебора, с использованием квадрата евклидова расстояния.

## Алгоритмы

### HNSW

**Библиотека:** `datasketch` (Python)

HNSW (Hierarchical Navigable Small World) - иерархический граф, в котором каждый вектор связан с несколькими ближайшими соседями на нескольких уровнях. Построение индекса происходит инкрементально: каждый новый вектор вставляется в граф, соединяясь с `m` ближайшими на каждом уровне. При поиске алгоритм «спускается» по уровням графа, на каждом шаге двигаясь к ближайшим известным соседям.

Ключевые параметры:


| Параметр          | Влияние                                                                                    |
| ----------------- | ------------------------------------------------------------------------------------------ |
| `m`               | Количество рёбер на вершину. Больше - выше точность и размер индекса, медленнее построение |
| `ef_construction` | Размер очереди при построении. Больше - точнее связи, медленнее вставка                    |
| `ef_query`        | Размер очереди при поиске. Больше - выше recall, ниже QPS                                  |


---

### LSH

**Библиотека:** собственная реализация (`RandomProjectionLSH`) на основе `numpy`

LSH (Locality Sensitive Hashing) основан на случайных проекциях: для каждого вектора генерируется бинарный хеш через проецирование на случайные гиперплоскости. Векторы с похожими хешами попадают в одни и те же «корзины». При запросе выбираются кандидаты из совпадающих корзин, а затем ранжируются по квадрату евклидова расстояния.

Ключевые параметры:


| Параметр   | Влияние                                                                       |
| ---------- | ----------------------------------------------------------------------------- |
| `n_planes` | Число бит в хеше. Больше - меньше коллизий, меньше кандидатов, ниже recall    |
| `n_tables` | Число независимых хеш-таблиц. Больше - шире охват, выше recall, больше памяти |


---

### IVF+PQ

**Библиотека:** `faiss` (Facebook AI Similarity Search, CPU-версия)

IVF+PQ (Inverted File Index + Product Quantization) - двухэтапный алгоритм. На первом этапе векторное пространство разбивается на `nlist` кластеров (IVF). При поиске исследуются только `nprobe` ближайших кластеров, что резко сокращает число сравнений. На втором этапе (PQ) каждый вектор сжимается: пространство делится на `m_pq` подпространств, и каждое кодируется отдельным квантователем. Это значительно уменьшает размер индекса.

Ключевые параметры:


| Параметр | Влияние                                                                                 |
| -------- | --------------------------------------------------------------------------------------- |
| `nlist`  | Число кластеров. Больше - тоньше разбивка, выше точность при том же `nprobe`            |
| `nprobe` | Число проверяемых кластеров. Больше - выше recall, ниже QPS                             |
| `m_pq`   | Число подпространств PQ. Фиксировано на 15 (300 / 15 = 20 измерений на подпространство) |

## Результаты

Метрики: Recall, Query QPS (запросов в секунду), Size (MB) - размер индекса в памяти. Индекс: 50 000 векторов.

Время построения и запроса измерялось через `time.perf_counter()`. Размер индекса определялся сериализацией: для FAISS использовался `faiss.serialize_index`, для HNSW и LSH - `pickle.dumps`; итоговый размер буфера в байтах переводился в мегабайты.

### Наблюдения по каждому алгоритму

**HNSW.** Recall растёт при увеличении `m` и `ef` - с 0.056 у `hnsw_1` до 0.369 у `hnsw_4`. Однако цена - на порядок более долгое построение (12.6 с - 143 с) и значительно больший индекс (~63–70 MB). QPS при поиске снижается пропорционально росту `ef_query`. Алгоритм хорошо управляем, но дорог и в построении, и в памяти.

**LSH.** Recall при `lsh_1` катастрофически мал (0.006), однако `lsh_4` с 32 таблицами даёт 0.288 - сопоставимо с лучшим HNSW. Индекс компактный (1–5 MB). Вместе с тем результаты показывают существенный изъян реализации: построение занимает 2.8–17.4 с - это неожиданно много для алгоритма, который теоретически должен строиться за доли секунды. Причина - использование чистого Python в цикле хеширования вместо векторизованных операций numpy.

**IVF+PQ.** Построение быстрее HNSW (2.3–2.7 с), индекс маленький (1.5–2 MB). Recall выходит на плато около 0.21 и почти не растёт при увеличении `nprobe` с 16 до 64 (0.211 - 0.217). Зато QPS при `ivfpq_1` достигает 16 000+ запросов в секунду при recall 0.153 - лучший показатель скорости среди всех алгоритмов при разумной точности.

## Вывод

Ни один из протестированных алгоритмов не является «лучшим» в абсолютном смысле - выбор зависит от приоритетов задачи.

В контексте данного датасета (300-мерные векторы, 50k индекс) **IVF+PQ** выглядит наиболее практичным компромиссом: он строится в десятки раз быстрее HNSW, занимает в 30–40 раз меньше памяти, а скорость запроса превосходит все остальные алгоритмы. Если же точность поиска критична и ресурсы позволяют, **HNSW** - предпочтительный выбор, так как его recall на ~70% выше лучшего результата IVF+PQ.

**LSH** в текущей реализации проигрывает обоим: медленное построение и низкий QPS объясняются Python-циклами в хешировании, а не самим алгоритмом. Идея LSH остаётся привлекательной - компактный индекс, простая логика, хорошо параллелизуется - но для конкурентоспособных показателей необходима низкоуровневая реализация (numpy-vectorized или нативный код).
